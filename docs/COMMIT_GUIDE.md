## 一、项目约束说明（强制）

> ⚠️ **重要提示**：本项目为 PNPM Monorepo + NestJS + Turbo 工程，以下约束必须严格遵守。

### 核心约束

- ✅ **仅可使用 `pnpm`**，禁止使用 `npm` / `npx` / `yarn`
- ✅ **禁止手写** `git commit -m` 提交
- ✅ **统一使用** `git cz` 交互式提交
- ✅ **本地提交**仅做增量格式化校验
- ✅ **远端 GitHub Actions** 做全量兜底校验（lint / type-check / build）

## 二、本地环境准备（首次拉代码必做）

### 1. 安装项目依赖

```bash
pnpm install
```

> 💡 自动执行 `husky install` 启用本地 Git 钩子。

### 2. 全局安装标准化提交工具（一次安装永久生效）

```bash
# 修复 pnpm 全局环境变量
pnpm setup

# 全局安装交互式提交工具
pnpm add -g git-cz
```

> ⚠️ **注意**：安装完成后需重启终端，之后在任意目录可直接使用 `git cz`。

## 三、标准完整提交流程（日常开发唯一流程）

### 📝 步骤概览

```mermaid
graph LR
    A[开发完成] --> B[git add .]
    B --> C[git cz]
    C --> D[自动校验]
    D --> E[git push]
    E --> F[CI 兜底校验]
```

### 详细步骤

#### 步骤 1：开发完成，确认代码无误

完成功能开发或 Bug 修复，准备提交代码。

#### 步骤 2：暂存变更文件

```bash
git add .
```

#### 步骤 3：规范交互式提交（核心）⭐

```bash
git cz
```

按照终端提示依次选择/输入：

1. **选择 commit type**（feat/fix/docs/refactor/style/chore/perf/test/revert）
2. **输入 scope 模块范围**（server / types / admin / utils / config）
3. **输入简短中文描述**（必填）
4. **详细描述、issue 关联**（可选，直接回车跳过）

#### 步骤 4：自动执行本地校验（无需手动操作）

提交瞬间自动触发 `pre-commit` 钩子：

- ✅ 执行 `pnpm dlx lint-staged`
- ✅ 仅对本次修改文件：ESLint 自动修复 + Prettier 格式化
- ❌ 存在无法自动修复的错误 → **阻断提交**，修复后重新提交

#### 步骤 5：提交完成，推送远端

```bash
git push
```

#### 步骤 6：远端 CI 自动兜底校验（强制卡点）

推送 `dev` / `main` 分支自动触发 GitHub Actions CI，执行全量校验：

- ✅ `pnpm lint` - 全局代码规范校验
- ✅ `pnpm type-check` - 全局 TS 类型校验
- ✅ `pnpm build` - 全项目打包编译验证

> ⚠️ **CI 不通过禁止合并 PR**，完全杜绝坏代码进入主干分支。

## 四、Commit Type 统一规范（团队严格统一）

| Type       | 说明                                        | 示例场景             |
| ---------- | ------------------------------------------- | -------------------- |
| `feat`     | 新增功能、接口、页面、模块                  | 新增用户登录接口     |
| `fix`      | 修复 Bug、逻辑异常、线上问题                | 修复订单计算错误     |
| `docs`     | 文档、注释、README 修改（无代码逻辑变动）   | 更新 API 文档        |
| `style`    | 代码格式化、空格、分号调整（无逻辑变更）    | 调整缩进格式         |
| `refactor` | 代码重构，无新增功能、无 Bug 修复           | 重构数据库查询逻辑   |
| `perf`     | 性能优化、打包优化、接口速度优化            | 优化图片加载速度     |
| `test`     | 新增/修改单元测试、E2E 测试                 | 添加用户服务测试用例 |
| `chore`    | 配置、依赖、脚本、husky、turbo、eslint 调整 | 升级依赖版本         |
| `revert`   | 回滚历史提交                                | 回滚错误提交         |

## 五、Scope 模块规范（本项目专用）

### 可用 Scope

| Scope    | 说明                       | 对应目录         |
| -------- | -------------------------- | ---------------- |
| `server` | Nest 后端服务              | `apps/backend`   |
| `types`  | 公共类型、DTO、枚举        | `packages/types` |
| `admin`  | 前端管理端                 | `apps/frontend`  |
| `utils`  | 公共工具函数               | `packages/utils` |
| `config` | 工程配置、CI、husky、turbo | 根目录配置文件   |

### 标准示例

```bash
feat(server): 新增用户分页查询接口
fix(types): 修复 Swagger DTO 类型丢失问题
chore(config): 优化 CI 缓存策略
refactor(server): 重构登录校验逻辑
docs(admin): 更新用户管理模块文档
```

## 六、禁止操作（团队红线）🚫

> ❌ **以下操作严格禁止，违者将影响代码审查结果**

- ❌ 禁止使用 `npm` / `npx` / `yarn` 任何命令
- ❌ 禁止手写 `git commit -m "xxx"`
- ❌ 禁止使用 `git commit --no-verify` 跳过本地校验
- ❌ 禁止直接 push 保护分支（必须 PR 合并）

## 七、本地自测全套命令（开发自测用，非提交触发）

```bash
# 全局代码规范检查
pnpm lint

# 全局 TS 类型检查
pnpm type-check

# 全量打包验证
pnpm build

# 一键自动修复所有代码格式问题
pnpm lint:fix
```

> 💡 **建议**：在提交前运行上述命令进行自测，可提前发现潜在问题。

## 八、整体流程总结

```
开发改代码
    ↓
git add 暂存
    ↓
git cz 规范提交
    ↓
自动 lint 格式化
    ↓
git push
    ↓
CI 全量兜底校验
    ↓
PR 评审合并
```

### 🎯 关键要点

1. **始终使用** `git cz` 进行提交
2. **本地校验**自动执行，无需手动干预
3. **CI 校验**是最后一道防线，确保代码质量
4. **遵守规范**是团队协作的基础
